# Nerd CI/CD Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `brainstorming-smart-execute`, which applies `superpowers:executing-plans` inline, to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a small, deterministic CI pipeline and a tag-driven delivery check that proves the public Nerd repository can be discovered and installed through Skills CLI.

**Architecture:** GitHub Actions owns two workflows. CI runs local syntax, unit, skill-contract, benchmark-plan, README-sync, and Skills CLI discovery checks on pull requests and default-branch pushes. CD runs the same release gate for version tags, then validates the immutable tag source and the public shorthand install in isolated agent homes. GitHub is the publication source; Skills CLI consumes that source.

**Tech Stack:** GitHub Actions, Python 3 standard library, Node.js 22, `unittest`, Skills CLI.

**Depends on:**

- `docs/plans/2026-07-15-nerd-skills.md` completed before local skill validation can pass.
- `docs/plans/2026-07-15-nerd-benchmarks.md` completed before benchmark-plan validation can pass.
- `docs/plans/2026-07-15-nerd-readme.md` completed before README synchronization can pass.
- The GitHub repository renamed to `Danangjoyoo/nerd`, public, and configured as `origin`.

**Implementation status (2026-07-15):** Both read-only workflows, their static contracts, and the README badge are implemented and pass locally. The first push/tag and remote workflow proof remain an operator step after the repository is public and the benchmark release gate is unblocked.

## Publication Semantics

- The correct install command is `npx skills add danangjoyoo/nerd`; `npx add danangjoyoo/nerd` is not a Skills CLI publication command.
- `npx skills add` installs or inspects skills from GitHub. It does not upload a package to skills.sh.
- Publishing means pushing the five valid skills to the public `Danangjoyoo/nerd` repository, then pushing a version tag.
- CD proves that Skills CLI can discover and install the public GitHub source. skills.sh discovery follows the GitHub-backed Skills CLI ecosystem; there is no separate repository upload step in this plan.
- The release workflow must verify both:
  - The immutable tag URL, so the exact release is installable.
  - The shorthand `danangjoyoo/nerd`, so the README command works from the default branch.
- This plan supersedes Task 5, “Add deterministic public CI,” in `docs/plans/2026-07-15-nerd-readme.md`. Do not implement both versions.

## Global Constraints

- Keep CI credential-free, deterministic, and free of paid live agent or judge calls.
- Use Python standard-library checks only; do not add a formatter, linter, YAML parser, or package manager solely for CI.
- Treat `python3 -m compileall` as the lightweight Python syntax/lint check and `unittest` as the behavior check.
- Pin GitHub Action majors and the Node/Python versions. Do not pin transient `npx` cache paths.
- Set workflow permissions to `contents: read`; CD does not create releases or mutate repository contents.
- Add concurrency cancellation for superseded branch/PR CI, but never cancel an in-progress tag validation.
- Add timeouts to every job.
- Never use `pull_request_target`, write tokens, repository secrets, `curl | sh`, or executable code from pull-request metadata.
- Never run `npx skills add` against a user home in CI; use local discovery or an isolated temporary `HOME`.
- Require exactly these five public skills:
  - `nerd-smart`
  - `nerd-surgery`
  - `nerd-patrol`
  - `nerd-execute`
  - `nerd-silent`
- Do not expose internal Superpowers-derived Markdown references as installable skills.

- Job runner: `ubuntu-latest`.
- Job timeout: 10 minutes.

- [x] **Step 3: Validate immutable tag discovery**
